# ADR-004: Teaching Suite 1.0 — Sprint 1 (Metadata + _config/)

**Status:** Proposed
**Date:** 2026-03-15
**Author:** Niklas + Claude (Cowork)

---

## Context

Teaching Suite v0.2.2 works in production (KURS101_2026) with 18 tools and 37 methodology documents. But files are isolated — no metadata connects them, no context loads automatically, no folder structure separates data types. The 1.0 vision (internal docs) defines the goal. This ADR decides *how* to take the first step.

## Options Considered

### Option A: Big bang — new folder structure + all metadata changes at once
- Pros: Everything in place immediately
- Cons: High risk, likely breaks existing tests, hard to roll back
- Effort: High

### Option B: Incremental — metadata first, then folder structure, then context loading
- Pros: Each step testable. Backward-compatible. 119 existing tests unaffected.
- Cons: Takes longer before the full picture is visible
- Effort: Medium (distributed)

### Option C: Only _config/ — create CLAUDE.md + course_context.md manually, rest later
- Pros: Fastest to value (context at session start)
- Cons: Metadata linkages missing, files remain isolated
- Effort: Low

## Decision

**Option B** — incremental, starting with `project_init` expansion:

1. `project_init` extended — new folder structure + `_config/` with templates
2. `intelligent_save` extended — metadata (course, based_on, supports)
3. `context_load` (new tool) — load _config/ at session start
4. `find_context` extended — search YAML fields + sources.yaml integration

Each step delivers value and can be tested independently. No existing tests shall break — all new fields are optional.

## Rationale

Incremental is the lowest risk. Teaching Suite is in production (KURS101_2026) — we must not break what works. Each iteration adds real functionality that is kept (evolutionary prototyping).

Walking skeleton approach: steps 1–3 give the thinnest end-to-end slice (init creates structure → save writes metadata → load reads metadata). Then built upon.

## Technical Decisions

### Metadata standard (YAML frontmatter)

Required fields for all files:
```yaml
---
type: reflection | lesson_plan | idea | analysis | insight | methodology
course: KURS101_2026
created: 2026-03-15
---
```

Optional fields (generated by intelligent_save):
```yaml
based_on:
  - Reflections/2026-03-14.md
supports: [G1, LO3]
framework: gibbs
learning_objectives: [LO1, LO3]
author: niklas
```

### _config/ files

- `course_context.md` — Course context with sections: Mission, Goals, Learning Objectives, Challenges, Journal
- `CLAUDE.md` — Per-course orchestration instructions
- `sources.yaml` — Source tracking (existing)
- `project_state.json` — Process state (existing)

### context_load — new tool

- Input: project workspace path
- Output: Combined context from CLAUDE.md + course_context.md + project_state.json + sources.yaml summary
- Usage: Called at session start. Returns structured context.
- Graceful degradation: returns empty context if _config/ missing

## Backward Compatibility

- All new metadata fields are **optional** — existing files without frontmatter work as before
- `project_init` creates new structure but **does not require it** — old course folders work
- `intelligent_save` adds metadata if course context exists, otherwise saves as today
- `context_load` returns empty context if `_config/` is missing
- All 119 existing tests shall **continue to pass**
